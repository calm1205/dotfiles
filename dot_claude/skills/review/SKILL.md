# 役割
未コミット変更またはPRに対する5観点並列コードレビューの実行

# ワークフロー

## 1. レビューモード判定・差分取得

### 引数によるモード分岐
- **PR番号/URLが引数に指定された場合** → PRモード
- **引数なし** → ローカル変更モード

### PRモード
- `gh pr diff <PR番号>` でPRの差分を取得
- `gh pr view <PR番号> --json title,body,baseRefName,headRefName` でPRのメタ情報を取得
- PRが存在しない・権限不足の場合はエラーメッセージを出力して終了

### ローカル変更モード
- `git diff` でステージ前の変更を取得
- `git diff --cached` でステージ済みの変更を取得
- `git diff HEAD` で未追跡ファイルを除く全変更を確認
- `git status` で未追跡ファイルの有無を確認し、レビュー対象に含める

### 共通
- 変更がなければ「レビュー対象なし」と出力して終了

## 2. 差分サイズの判定
- 差分が小さい場合（概ね100行以下）: エージェント1つで5観点を統合レビュー
- 差分が大きい場合（概ね100行超）: 5エージェントで並列レビュー（ステップ3へ）

## 3. チーム作成・並列レビュー
- TeamCreateでレビューチームを作成
- 以下5エージェントをAgentツールで**並列起動**（各エージェントは `subagent_type: "general-purpose"` を使用）
- 各エージェントには差分をプロンプトに含めて渡す

### エージェント構成

#### 1. security-reviewer - セキュリティ脆弱性の検出
- OWASP Top 10:2025に基づく脆弱性チェック
- インジェクション（SQL、コマンド、XSS）
- 認証・認可の漏れ（エンドポイントの認証有無、他テナントデータへのアクセス）
- 機密情報の露出（ハードコードされた秘密鍵、ログへの機密情報出力）
- 安全でないデシリアライズ、SSRF

#### 2. architecture-reviewer - 設計・規約準拠の検証
- CLAUDE.mdおよび`.claude/rules/`配下のプロジェクト規約への準拠
- 責務分離（単一責任原則）
- 依存関係の方向性（クリーンアーキテクチャの層構造に沿っているか）
- 既存コードとのパターン整合性

#### 3. logic-reviewer - バグ・ロジックエラーの検出
- nullポインタ、off-by-one、型不一致などの典型バグ
- エッジケースの未処理（空配列、空文字、boundary値）
- エラーハンドリングの抜け（try/except漏れ、rollback漏れ）
- N+1クエリ、不要なDB呼び出し
- 競合状態、データ不整合のリスク

#### 4. test-reviewer - テスト品質の検証
- テストケースの網羅性
  - 異常系: リソース不存在、削除済み、他テナント、必須パラメータ欠落
  - 正常系: 主要パスの検証
- テストコードが変更されていない場合: 変更内容に対してテスト追加が必要かを指摘

#### 5. tradeoff-analyst - トレードオフの可視化
- 変更が「何を得て、何を犠牲にしているか」を分析し、ユーザーに問いかける
- 指摘ではなく**問い**として出力する（ユーザーが意図を確認するため）
- 分析対象の軸:
  - パフォーマンス ↔ 可読性・保守性
  - 柔軟性・拡張性 ↔ シンプルさ
  - 型安全性・堅牢性 ↔ 開発速度
  - DRY（共通化） ↔ 明示性・独立性
  - 即時性（同期処理） ↔ スケーラビリティ（非同期処理）
- 出力形式（他エージェントと異なる）:
  ```
  ## トレードオフ
  - **得たもの**: 〇〇
    **犠牲**: △△
    **問い**: この変更で△△が犠牲になっていますが、意図通りですか？ [file:line]
  ```
- トレードオフが検出されない場合は「トレードオフなし」と返す

### 各エージェント共通の指示
- 差分のみをレビュー対象とする（差分外のコードは参考程度）
- 必要に応じてGlob/Grep/Readで周辺コードを確認
- 推測ではなくコードの事実に基づいて指摘する
- false positiveを避け、確信度の高い指摘に絞る
- 自分の担当観点のみ指摘する（他エージェントの観点と重複しない）
- 結果は以下の形式で返すこと:
  ```
  ## Critical（修正必須）
  - 説明 [file:line]

  ## Warning（推奨修正）
  - 説明 [file:line]

  ## Info（参考）
  - 説明 [file:line]
  ```
- 指摘がないカテゴリは省略可
- 指摘がゼロの場合は「指摘なし」と返す

## 4. 結果集約・出力
- 全エージェントの結果を待機
- チームを削除
- 重複する指摘を排除し、重要度順に並べ替え
- 以下の形式でターミナルに出力:

```markdown
# レビュー結果

## Critical（修正必須）
- [セキュリティ/A01]: 説明 [file:line]
- [ロジック]: 説明 [file:line]

## Warning（推奨修正）
- [設計]: 説明 [file:line]
- [テスト]: 説明 [file:line]

## Info（参考）
- [設計]: 説明 [file:line]

## トレードオフ
- **得たもの**: 〇〇
  **犠牲**: △△
  **問い**: この変更で△△が犠牲になっていますが、意図通りですか？ [file:line]
```

# ルール
- 全エージェントの完了を待ってから結果を出力
- 指摘がないカテゴリ（Critical/Warning/Info）は省略
- 全観点で指摘なしの場合は「問題なし」と出力
- レビュー結果の出力は日本語で
- 各指摘には必ずファイルパスと行番号を含める
- 各指摘には観点タグを付記: [セキュリティ], [設計], [ロジック], [テスト], [トレードオフ]
- トレードオフは指摘ではなく問いかけとして出力（Critical/Warning/Infoとは別セクション）
- セキュリティ指摘にはOWASPカテゴリ番号（A01〜A10）を付記
- PRモード時はレビュー結果ヘッダーにPR番号・タイトルを表示
