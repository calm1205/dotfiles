# 役割
未コミット変更またはPRに対する5観点並列コードレビューのオーケストレーション

# ワークフロー

## 1. 差分取得
- **PR番号/URLが引数にある場合** → `gh pr diff` + `gh pr view --json title,body,baseRefName,headRefName`
- **引数なし** → `git diff HEAD` + `git status`（未追跡ファイル含む）
- 変更がなければ「レビュー対象なし」と出力して終了

## 2. 差分サイズの判定
- テストファイル（`*_test.*`, `*.test.*`, `*.spec.*`, `test_*`, `tests/`, `__tests__/`）の差分行数を除外して判定
- テスト以外の修正が100行以下 → 単一エージェント（`architecture-reviewer`）で統合レビュー
- 100行超 → 5エージェントで並列レビュー（ステップ3へ）

## 3. 並列レビュー
以下5エージェントをAgentツールで**並列起動**。各エージェントには取得済みの差分をプロンプトに含めて渡す。

| サブエージェント | 観点 |
|---|---|
| `security-reviewer` | OWASP Top 10:2025に基づくセキュリティ脆弱性 |
| `architecture-reviewer` | CLAUDE.md・規約準拠、責務分離、既存パターン整合性 |
| `logic-reviewer` | バグ・エッジケース・エラーハンドリング漏れ・N+1等 |
| `test-reviewer` | テスト網羅性。未変更ならテスト追加の必要性を指摘 |
| `tradeoff-analyst` | 変更が何を得て何を犠牲にしたかを問いとして提示 |

### プロンプト共通指示
- 差分のみをレビュー対象（必要に応じてGlob/Grep/Readで周辺確認）
- 推測ではなくコードの事実に基づき、確信度の高い指摘に絞る
- 自分の担当観点のみ指摘（他観点と重複しない）
- 指摘ゼロなら「指摘なし」と返す

## 4. 結果集約・出力
全エージェント完了後、重複排除・重要度順に並べ替えて出力。

```markdown
# レビュー結果

## Critical（修正必須）
- [セキュリティ]: 指摘事項 [file:line]
  - 指摘理由
  - 修正提案

## Warning（推奨修正）
- [設計]: 指摘事項 [file:line]
  - 指摘理由
  - 修正提案

## トレードオフ
- **得たもの**: 〇〇 [file:line]
  **犠牲**: △△
  **問い**: 許容範囲か？
```

# ルール
- 指摘がないカテゴリは省略。全観点で指摘なしなら「問題なし」
- 各指摘に観点タグ（[セキュリティ], [設計], [ロジック], [テスト]）とファイルパス:行番号を付記
- PRモード時はヘッダーにPR番号・タイトルを表示
