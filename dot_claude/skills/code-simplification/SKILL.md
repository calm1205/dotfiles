# コード簡素化（Code Simplification）

## 概要

挙動を厳密に保ったまま複雑さを減らすことでコードを簡素化する。
目標は行数を減らすことではなく、読みやすく・理解しやすく・変更しやすく・デバッグしやすいコードにすること。
すべての簡素化は次のシンプルなテストを通過しなければならない。
「新しいチームメンバーは、元のコードより速く理解できるか？」

## 使用すべきとき

- 機能が動作しテストも通ったが、実装が必要以上に重く感じるとき
- コードレビューで可読性や複雑さの問題が指摘されたとき
- 深いネストのロジック、長すぎる関数、不明瞭な名前に遭遇したとき
- 時間的プレッシャーの中で書かれたコードをリファクタリングするとき
- 重複や不整合を持ち込んだ変更をマージした後

**使用すべきでないとき:**

- コードがすでにクリーンで読みやすい — 簡素化のための簡素化はしない
- コードが何をしているのかまだ理解していない — 簡素化の前に理解する
- コードが性能上クリティカルで、「より簡潔な」版が計測上明らかに遅くなる
- モジュールごと書き直そうとしている — 使い捨てコードの簡素化は労力の無駄

## 5つの原則

### 1. 挙動を厳密に保つ

コードが「何をするか」は変えず、「どう表現するか」だけを変える。
すべての入力・出力・副作用・エラー挙動・エッジケースは同一のまま保たれなければならない。
挙動を保つと確信できない簡素化なら、行わない。

```
すべての変更の前に問う:
→ すべての入力に対して同じ出力を生むか？
→ 同じエラー挙動を保つか？
→ 同じ副作用と順序を保つか？
→ 既存テストはすべて修正なしで通るか？
```

### 2. プロジェクトの慣習に従う

簡素化とは、外部の好みを押し付けることではなく、コードをコードベースとより一貫させること。
簡素化の前に:

```
1. CLAUDE.md / プロジェクト慣習を読む
2. 近隣のコードが類似パターンをどう扱っているか調べる
3. プロジェクトのスタイルに合わせる対象:
   - import の並び順とモジュールシステム
   - 関数宣言のスタイル
   - 命名規則
   - エラーハンドリングのパターン
   - 型アノテーションの粒度
```

プロジェクトの一貫性を壊す簡素化は、簡素化ではない — それは無駄な変更（churn）。

### 3. 賢さより明瞭さを優先する

明示的なコードは、コンパクトな版がパースに一拍の思考を要するなら、コンパクトなコードより優れている。

```typescript
// 不明瞭: 密な三項演算子の連鎖
const label = isNew ? 'New' : isUpdated ? 'Updated' : isArchived ? 'Archived' : 'Active';

// 明瞭: 読みやすいマッピング
function getStatusLabel(item: Item): string {
  if (item.isNew) return 'New';
  if (item.isUpdated) return 'Updated';
  if (item.isArchived) return 'Archived';
  return 'Active';
}
```

```typescript
// 不明瞭: インラインロジックを伴う reduce の連鎖
const result = items.reduce((acc, item) => ({
  ...acc,
  [item.id]: { ...acc[item.id], count: (acc[item.id]?.count ?? 0) + 1 }
}), {});

// 明瞭: 名前付きの中間ステップ
const countById = new Map<string, number>();
for (const item of items) {
  countById.set(item.id, (countById.get(item.id) ?? 0) + 1);
}
```

### 4. バランスを保つ

簡素化には失敗モードがある。
過剰な簡素化だ。
次の罠に注意する。

- **インライン化しすぎ** — 概念に名前を与えていたヘルパーを除去すると、呼び出し側が読みにくくなる
- **無関係なロジックの結合** — 2つの単純な関数を1つの複雑な関数にまとめても簡潔ではない
- **「不要な」抽象の除去** — 一部の抽象は複雑さのためではなく、拡張性やテスト容易性のために存在する
- **行数の最適化** — 行数の少なさは目標ではない。理解しやすさが目標

### 5. 変更箇所にスコープを限定する

デフォルトでは、最近変更したコードのみを簡素化する。
明示的に依頼されない限り、無関係なコードの「ついでリファクタ」は避ける。
スコープ外の簡素化は差分にノイズを生み、意図しない退行のリスクを伴う。

## 簡素化のプロセス

### ステップ1: 触れる前に理解する（チェスタートンの柵）

何かを変更・除去する前に、それがなぜ存在するのかを理解する。
これがチェスタートンの柵だ。
道を塞ぐ柵を見て、なぜそこにあるのか分からないなら、取り壊してはならない。
まず理由を理解し、その理由がまだ当てはまるかを判断する。

```
簡素化の前に答える:
- このコードの責務は何か？
- 何がこれを呼び、これは何を呼ぶか？
- エッジケースとエラー経路は何か？
- 期待される挙動を定義するテストはあるか？
- なぜこう書かれた可能性があるか？（性能？ プラットフォーム制約？ 歴史的理由？）
- git blame を確認する: このコードの元の文脈は何だったか？
```

これらに答えられないなら、簡素化の準備はできていない。
先に文脈をもっと読む。

### ステップ2: 簡素化の機会を特定する

次のパターンを走査する — それぞれが具体的なシグナルであり、漠然とした「におい」ではない。

**構造的な複雑さ:**

| パターン | シグナル | 簡素化 |
|---------|--------|----------------|
| 深いネスト（3階層以上） | 制御フローを追いにくい | 条件をガード節やヘルパー関数に抽出 |
| 長い関数（50行以上） | 複数の責務 | 説明的な名前を持つ焦点の絞られた関数に分割 |
| ネストした三項演算子 | パースに脳内スタックが必要 | if/else 連鎖、switch、ルックアップオブジェクトに置換 |
| 真偽値のパラメータフラグ | `doThing(true, false, true)` | オプションオブジェクトや個別の関数に置換 |
| 繰り返される条件 | 複数箇所で同じ `if` チェック | 名前の付いた述語関数に抽出 |

**命名と可読性:**

| パターン | シグナル | 簡素化 |
|---------|--------|----------------|
| 一般的すぎる名前 | `data`, `result`, `temp`, `val`, `item` | 中身を表す名前に変更: `userProfile`, `validationErrors` |
| 省略された名前 | `usr`, `cfg`, `btn`, `evt` | 省略が普遍的（`id`, `url`, `api`）でない限り完全な単語を使う |
| 誤解を招く名前 | `get` という名前なのに状態も変更する関数 | 実際の挙動を反映した名前に変更 |
| 「何を」を説明するコメント | `count++` の上の `// increment counter` | コメントを削除 — コードが十分に明瞭 |
| 「なぜ」を説明するコメント | `// 負荷時に API が不安定なためリトライ` | 残す — コードでは表現できない意図を運んでいる |

**冗長性:**

| パターン | シグナル | 簡素化 |
|---------|--------|----------------|
| 重複したロジック | 複数箇所に同じ5行以上 | 共有関数に抽出 |
| デッドコード | 到達不能な分岐、未使用変数、コメントアウトされたブロック | （本当に死んでいると確認のうえ）削除 |
| 不要な抽象 | 価値を加えないラッパー | ラッパーをインライン化し、下層関数を直接呼ぶ |
| 過剰設計のパターン | ファクトリのためのファクトリ、単一戦略のストラテジ | シンプルな直接的アプローチに置換 |
| 冗長な型アサーション | すでに推論済みの型へのキャスト | アサーションを削除 |

### ステップ3: 変更を漸進的に適用する

一度に1つずつ簡素化する。
各変更の後にテストを実行する。
**リファクタリングの変更は、機能追加やバグ修正とは別に提出する。**
リファクタと機能追加が混ざった PR は2つの PR だ。
分割する。

```
各簡素化について:
1. 変更を行う
2. テストスイートを実行する
3. テスト通過 → コミット（または次の簡素化へ進む）
4. テスト失敗 → 元に戻して再考する
```

複数の簡素化を1つの未テスト変更にまとめない。
何かが壊れたとき、どの簡素化が原因かを知る必要がある。

**500の法則:** リファクタリングが500行を超えて触れる場合、手作業ではなく自動化（codemod、sed スクリプト、AST 変換）に投資する。
その規模での手作業は誤りを生みやすく、レビューも消耗する。

### ステップ4: 結果を検証する

すべての簡素化の後、一歩引いて全体を評価する。

```
ビフォーとアフターを比較する:
- 簡素化した版は本当に理解しやすいか？
- コードベースと不整合な新しいパターンを持ち込んでいないか？
- 差分はクリーンでレビュー可能か？
- チームメイトはこの変更を承認するか？
```

「簡素化した」版が理解しにくく、レビューしにくいなら、元に戻す。
すべての簡素化の試みが成功するわけではない。

## 言語別ガイダンス

### TypeScript / JavaScript

```typescript
// 簡素化: 不要な async ラッパー
// ビフォー
async function getUser(id: string): Promise<User> {
  return await userService.findById(id);
}
// アフター
function getUser(id: string): Promise<User> {
  return userService.findById(id);
}

// 簡素化: 冗長な条件代入
// ビフォー
let displayName: string;
if (user.nickname) {
  displayName = user.nickname;
} else {
  displayName = user.fullName;
}
// アフター
const displayName = user.nickname || user.fullName;

// 簡素化: 手動の配列構築
// ビフォー
const activeUsers: User[] = [];
for (const user of users) {
  if (user.isActive) {
    activeUsers.push(user);
  }
}
// アフター
const activeUsers = users.filter((user) => user.isActive);

// 簡素化: 冗長な真偽値の return
// ビフォー
function isValid(input: string): boolean {
  if (input.length > 0 && input.length < 100) {
    return true;
  }
  return false;
}
// アフター
function isValid(input: string): boolean {
  return input.length > 0 && input.length < 100;
}
```

```python
# 簡素化: 冗長な辞書構築

result = {}

    result[item.id] = item.name

# ビフォー
def process(data):
    if data is not None:
        if data.is_valid():
            if data.has_permission():
                return do_work(data)
            else:
                raise PermissionError("No permission")
        else:
            raise ValueError("Invalid data")
    else:
        raise TypeError("Data is None")
# アフター
def process(data):
    if data is None:
        raise TypeError("Data is None")
    if not data.is_valid():
        raise ValueError("Invalid data")
    if not data.has_permission():
        raise PermissionError("No permission")
    return do_work(data)
```

## よくある自己正当化

| 自己正当化 | 現実 |
|---|---|
| 「動いてるんだから触る必要ない」 | 読みにくい動作コードは、壊れたときに直しにくい。今簡素化すれば、将来のすべての変更で時間を節約できる。 |
| 「行数が少ないほど常に簡潔」 | 1行のネストした三項演算子は、5行の if/else より簡潔ではない。簡潔さは理解の速さであり、行数ではない。 |
| 「ついでにこの無関係なコードもサッと簡素化しよう」 | スコープ外の簡素化はノイズの多い差分を生み、変更するつもりのなかったコードに退行リスクをもたらす。焦点を保つ。 |
| 「型が自己文書化してくれる」 | 型は構造を文書化するが、意図はしない。名前の付いた関数は、型シグネチャが「何を」を説明する以上に「なぜ」をうまく説明する。 |
| 「この抽象は後で役立つかも」 | 投機的な抽象を残さない。今使われていないなら、価値のない複雑さ。除去し、必要になったら再追加する。 |
| 「元の作者には理由があったはず」 | あるかもしれない。git blame を確認し、チェスタートンの柵を適用する。だが蓄積した複雑さには理由がないことも多い。それはプレッシャー下の反復が残した残滓にすぎない。 |
| 「機能追加のついでにリファクタする」 | リファクタリングは機能作業と分離する。混在した変更は、レビュー・取り消し・履歴理解が難しくなる。 |

## レッドフラグ

- テストを修正しないと通らない簡素化（おそらく挙動を変えている）
- 元より長く追いにくい「簡素化された」コード
- プロジェクト慣習ではなく自分の好みに合わせた名前変更
- 「コードがクリーンになるから」という理由でのエラーハンドリング除去
- 完全には理解していないコードの簡素化
- 多数の簡素化を1つの大きくレビューしにくいコミットにまとめる
- 依頼されていないのに現タスクのスコープ外のコードをリファクタリングする

## 検証

簡素化パスの完了後:

- [ ] 既存テストがすべて修正なしで通る
- [ ] ビルドが成功し、新しい警告が出ない
- [ ] リンタ/フォーマッタが通る（スタイルの退行なし）
- [ ] 各簡素化がレビュー可能な漸進的変更になっている
- [ ] 差分がクリーン — 無関係な変更が混ざっていない
- [ ] 簡素化したコードがプロジェクト慣習に従う（CLAUDE.md 等と照合済み）
- [ ] エラーハンドリングを除去・弱体化していない
- [ ] デッドコードを残していない（未使用 import、到達不能分岐）
- [ ] チームメイトやレビューエージェントが正味の改善として承認する
