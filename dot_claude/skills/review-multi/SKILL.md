# コードレビューと品質（Code Review and Quality）

## 概要

品質ゲートを伴う多次元コードレビュー。すべての変更はマージ前にレビューされる — 例外なし。レビューは5つの軸を網羅する。正確性・可読性・アーキテクチャ・セキュリティ・性能。

**承認の基準:** 完璧でなくとも、コード全体の健全性を確実に向上させるなら、その変更を承認する。完璧なコードは存在しない — 目標は継続的な改善。自分が書いたであろう書き方と完全には一致しないという理由で変更をブロックしない。コードベースを改善し、プロジェクトの慣習に従っているなら、承認する。

## 使用すべきとき

- あらゆる PR や変更のマージ前
- 機能実装の完了後
- 他のエージェントやモデルが生成したコードを評価する必要があるとき
- 既存コードのリファクタリング時
- あらゆるバグ修正の後（修正と退行テストの両方をレビュー）

## 5軸レビュー

すべてのレビューは次の次元でコードを評価する。

### 1. 正確性（Correctness）

コードは主張どおりのことをするか？

- 仕様やタスク要件に合致しているか？
- エッジケースは処理されているか（null、空、境界値）？
- エラー経路は処理されているか（ハッピーパスだけでなく）？
- すべてのテストを通るか？ テストは正しいものを実際に検証しているか？
- オフバイワンエラー、競合状態、状態の不整合はないか？

### 2. 可読性と簡潔性（Readability & Simplicity）

別のエンジニア（やエージェント）が、作者の説明なしにこのコードを理解できるか？

- 名前は説明的で、プロジェクト慣習と一貫しているか？（文脈のない `temp`, `data`, `result` はないか）
- 制御フローは素直か（ネストした三項演算子、深いコールバックを避ける）？
- コードは論理的に構成されているか（関連コードがまとまり、モジュール境界が明確）？
- 簡素化すべき「賢い」トリックはないか？
- **より少ない行数でできないか？**（100行で足りるのに1000行は失敗）
- **抽象は複雑さに見合っているか？**（3つ目のユースケースまで一般化しない）
- 自明でない意図を明確にするコメントは役立つか？（ただし自明なコードにコメントしない）
- デッドコードの痕跡はないか: 無意味な変数（`_unused`）、後方互換のシム、`// removed` コメント

### 3. アーキテクチャ（Architecture）

変更はシステムの設計に合致するか？

- 既存パターンに従うか、新しいパターンを導入するか？ 新規なら、正当化できるか？
- クリーンなモジュール境界を保つか？
- 共有すべきコードの重複はないか？
- 依存は正しい方向に流れているか（循環依存なし）？
- 抽象レベルは適切か（過剰設計でなく、密結合でもない）？

### 4. セキュリティ（Security）

詳細なセキュリティガイダンスは `security-and-hardening` を参照。変更は脆弱性を持ち込むか？

- ユーザー入力は検証・サニタイズされているか？
- 秘密情報はコード・ログ・バージョン管理の外に保たれているか？
- 必要な箇所で認証・認可がチェックされているか？
- SQL クエリはパラメータ化されているか（文字列連結なし）？
- 出力は XSS 防止のためエンコードされているか？
- 依存は信頼できる出所で、既知の脆弱性がないか？
- 外部ソース（API、ログ、ユーザーコンテンツ、設定ファイル）のデータは信頼できないものとして扱われているか？
- 外部データフローはロジックやレンダリングに使う前にシステム境界で検証されているか？

### 5. 性能（Performance）

詳細なプロファイリングと最適化は `performance-optimization` を参照。変更は性能問題を持ち込むか？

- N+1 クエリのパターンはないか？
- 無制限のループや無制約のデータ取得はないか？
- 非同期にすべき同期処理はないか？
- UI コンポーネントに不要な再レンダリングはないか？
- リストエンドポイントにページネーションの欠落はないか？
- ホットパスで大きなオブジェクトを生成していないか？

## 変更のサイズ

小さく焦点の絞られた変更は、レビューしやすく、マージが速く、デプロイが安全。次のサイズを目標にする。

```
~100行の変更   → 良好。一度で読みきれる。
~300行の変更   → 単一の論理的変更なら許容。
~1000行の変更  → 大きすぎる。分割する。
```

**「1つの変更」とは:** 1つのことに対処し、関連テストを含み、提出後もシステムを機能させる、単一で自己完結した修正。機能全体ではなく、機能の一部分。

**変更が大きすぎるときの分割戦略:**

| 戦略 | やり方 | いつ |
|----------|-----|------|
| **スタック** | 小さな変更を提出し、それを土台に次を始める | 逐次的な依存があるとき |
| **ファイルグループ別** | 異なるレビュアーを要するグループごとに変更を分ける | 横断的関心事 |
| **水平** | 共有コード/スタブを先に作り、次に利用側 | 階層アーキテクチャ |
| **垂直** | 機能をより小さなフルスタックのスライスに分ける | 機能開発 |

**大きな変更が許容されるとき:** ファイル全体の削除や、レビュアーが各行ではなく意図のみ検証すればよい自動リファクタリング。

**リファクタリングは機能作業と分離する。** 既存コードをリファクタしつつ新しい挙動を追加する変更は、2つの変更だ — 別々に提出する。小さな整理（変数名の変更）はレビュアーの裁量で含めてよい。

## 変更の説明

すべての変更には、バージョン管理履歴の中で単独で意味をなす説明が必要。

**1行目:** 短く、命令形で、単独で完結。「Deleting the FizzBuzz RPC」ではなく「Delete the FizzBuzz RPC」。履歴を検索する人が diff を読まずに変更を理解できるだけの情報量を持つこと。

**本文:** 何が、なぜ変わるか。コード自体には見えない文脈・判断・理由を含める。関連するバグ番号・ベンチマーク結果・設計ドキュメントへリンクする。アプローチの欠点があれば率直に認める。

**アンチパターン:** 「Fix bug」「Fix build」「Add patch」「Moving code from A to B」「Phase 1」「Add convenience functions」。

## レビュープロセス

### ステップ1: 文脈を理解する

コードを見る前に、意図を理解する。

```
- この変更は何を達成しようとしているか？
- どの仕様やタスクを実装するか？
- 期待される挙動の変化は何か？
```

### ステップ2: 先にテストをレビューする

テストは意図とカバレッジを明らかにする。

```
- 変更に対するテストは存在するか？
- 挙動を（実装詳細ではなく）テストしているか？
- エッジケースはカバーされているか？
- テストは説明的な名前を持つか？
- コードが変わったとき、テストは退行を捕捉できるか？
```

### ステップ3: 実装をレビューする

5つの軸を念頭にコードを通読する。

```
変更された各ファイルについて:
1. 正確性: このコードはテストが言うとおりに動くか？
2. 可読性: 助けなしに理解できるか？
3. アーキテクチャ: システムに合致するか？
4. セキュリティ: 脆弱性はないか？
5. 性能: ボトルネックはないか？
```

### ステップ4: 指摘を分類する

すべてのコメントに重大度のラベルを付け、必須か任意かを作者に伝える。

| 接頭辞 | 意味 | 作者の対応 |
|--------|---------|---------------|
| *(接頭辞なし)* | 必須の変更 | マージ前に必ず対処 |
| **Critical:** | マージをブロック | セキュリティ脆弱性、データ損失、機能破壊 |
| **Nit:** | 軽微・任意 | 作者は無視可 — フォーマット、スタイルの好み |
| **Optional:** / **Consider:** | 提案 | 検討の価値はあるが必須ではない |
| **FYI** | 情報のみ | 対応不要 — 将来のための文脈 |

これにより、作者がすべてのフィードバックを必須と捉え、任意の提案に時間を浪費するのを防ぐ。

### ステップ5: 検証を検証する

作者の検証ストーリーを確認する。

```
- どんなテストを実行したか？
- ビルドは通ったか？
- 手動でテストされたか？
- UI 変更にスクリーンショットはあるか？
- ビフォー/アフターの比較はあるか？
```

## マルチモデルレビューのパターン

レビュー観点ごとに異なるモデルを使う。

```
モデルA がコードを書く
    │
    ▼
モデルB が正確性とアーキテクチャをレビュー
    │
    ▼
モデルA がフィードバックに対処
    │
    ▼
人間が最終判断を下す
```

これは単一モデルが見逃しうる問題を捕捉する — モデルごとに盲点が異なる。

**レビューエージェント向けプロンプト例:**
```
このコード変更を、正確性・セキュリティ・我々のプロジェクト慣習への準拠の
観点でレビューせよ。仕様は [X]。変更は [Y] すべき。
問題は Critical / Required / Optional / Nit に分類して指摘せよ。
```

## デッドコードの衛生

リファクタリングや実装変更の後、孤立したコードを確認する。

1. 到達不能・未使用になったコードを特定する
2. それを明示的にリストする
3. **削除前に確認する:** 「これらの未使用要素を今削除すべきか: [リスト]？」

デッドコードを放置しない — 将来の読み手やエージェントを混乱させる。だが確信のないものを黙って削除しない。迷ったら確認する。

```
特定されたデッドコード:
- src/utils/date.ts の formatLegacyDate() — formatDate() で置換済み
- src/components/ の OldTaskCard コンポーネント — TaskCard で置換済み
- src/config.ts の LEGACY_API_URL 定数 — 残存参照なし
→ これらを削除して安全か？
```

## 意見の相違の扱い

レビューの論争を解決するとき、次の階層を適用する。

1. **技術的事実とデータ**は意見や好みに優先する
2. **スタイルガイド**はスタイル事項における絶対的権威
3. **ソフトウェア設計**は個人の好みではなく工学的原則で評価する
4. **コードベースの一貫性**は全体の健全性を損なわない限り許容される

**「後で整理する」を受け入れない。** 経験上、先送りした整理はめったに実行されない。真の緊急時を除き、提出前の整理を要求する。周辺の問題をこの変更で扱えないなら、自己割り当てでバグ起票を要求する。

## レビューにおける誠実さ

コードをレビューするとき — 自分・他のエージェント・人間のいずれが書いたものであれ:

- **ゴム印を押さない。** レビューの証拠のない「LGTM」は誰の役にも立たない。
- **本当の問題を和らげない。** 本番を直撃するバグを「些細な懸念かもしれません」と言うのは不誠実。
- **可能なら問題を定量化する。** 「この N+1 クエリはリストの項目ごとに約50ms 追加する」は「遅くなるかも」より良い。
- **明らかな問題のあるアプローチには押し返す。** 迎合はレビューにおける失敗モード。実装に問題があるなら、直接そう言い、代替案を提示する。
- **覆されることを潔く受け入れる。** 作者が完全な文脈を持ち異論があるなら、その判断に委ねる。人ではなくコードにコメントする — 個人批判はコード自体に焦点を当て直す。

## 依存の規律

コードレビューの一部は依存のレビュー。

**依存を追加する前に:**
1. 既存のスタックで解決できるか？（しばしば解決できる）
2. 依存はどれほど大きいか？（バンドルへの影響を確認）
3. 活発にメンテされているか？（最終コミット、未解決 issue を確認）
4. 既知の脆弱性はないか？（`npm audit`）
5. ライセンスは？（プロジェクトと互換でなければならない）

**ルール:** 新しい依存より、標準ライブラリと既存ユーティリティを優先する。あらゆる依存は負債。

## レビューチェックリスト

```markdown
## レビュー: [PR/変更タイトル]

### 文脈
- [ ] この変更が何をし、なぜ行うのか理解した

### 正確性
- [ ] 変更が仕様/タスク要件に合致
- [ ] エッジケースを処理
- [ ] エラー経路を処理
- [ ] テストが変更を十分にカバー

### 可読性
- [ ] 名前が明確で一貫
- [ ] ロジックが素直
- [ ] 不要な複雑さがない

### アーキテクチャ
- [ ] 既存パターンに従う
- [ ] 不要な結合や依存がない
- [ ] 適切な抽象レベル

### セキュリティ
- [ ] コードに秘密情報がない
- [ ] 境界で入力を検証
- [ ] インジェクション脆弱性がない
- [ ] 認可チェックが配置済み
- [ ] 外部データソースを信頼できないものとして扱う

### 性能
- [ ] N+1 パターンがない
- [ ] 無制限の処理がない
- [ ] リストエンドポイントにページネーション

### 検証
- [ ] テストが通る
- [ ] ビルドが成功
- [ ] 手動検証済み（該当時）

### 判定
- [ ] **承認** — マージ可
- [ ] **変更要求** — 問題への対処が必要
```

## よくある自己正当化

| 自己正当化 | 現実 |
|---|---|
| 「動いてるんだから十分」 | 読めない・安全でない・設計の誤った動作コードは、複利で膨らむ負債を生む。 |
| 「自分が書いたから正しいと分かる」 | 作者は自分の前提に盲目。あらゆる変更は別の目から恩恵を受ける。 |
| 「後で整理する」 | 後はやって来ない。レビューが品質ゲートだ — 使え。整理はマージ後ではなくマージ前に要求する。 |
| 「AI生成コードはたぶん大丈夫」 | AIコードには、少なくではなく、より多くの精査が必要。誤っていても自信ありげで尤もらしい。 |
| 「テストが通るから良い」 | テストは必要だが十分ではない。アーキテクチャ問題・セキュリティ問題・可読性の懸念は捕捉しない。 |

## レッドフラグ

- レビューなしでマージされた PR
- テストが通るかだけを確認するレビュー（他の軸を無視）
- 実際のレビューの証拠のない「LGTM」
- セキュリティに敏感な変更を、セキュリティ重点のレビューなしで実施
- 「大きすぎて適切にレビューできない」大きな PR（分割せよ）
- バグ修正 PR に退行テストがない
- 重大度ラベルのないレビューコメント — 必須か任意か不明瞭になる
- 「後で直す」を受け入れる — 実現しない

## 検証

レビュー完了後:

- [ ] すべての Critical 問題が解決済み
- [ ] すべての Required（接頭辞なし）変更が解決、または正当化付きで明示的に先送り
- [ ] テストが通る
- [ ] ビルドが成功
- [ ] 検証ストーリーが文書化されている（何が変わり、どう検証したか）
