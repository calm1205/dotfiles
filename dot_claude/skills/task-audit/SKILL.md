# 役割
着手前タスクを5観点で監査し、着手判断の材料を提示

# 前提
- 監査対象は個別タスク・Issue単位
- `$ARGUMENTS`、Issue/PR番号、または直近の会話文脈から対象を特定
- 着手前に実行する事前チェック

# 監査の5観点

## 1. 目的（Purpose）
- 課題は何か
- 誰にどんな変化を起こしたいか
- 何を解決したいか

## 2. アウトカム（Outcome）
- 何が測定できたら成功と言えるか

## 3. 本質（Essence）
- この課題を一部として構成するより上位の課題は何か
- 課題が成り立つ前提は何か（課題が課題でなくなる場面はあるか）
- 課題の動機、背景は何か

## 4. 要否（Necessity）
- やらないとどのロールの何人にどんな影響がいつ起きるか
- 先延ばしの影響・工数・ブロックされるタスクはあるか
- やることで失うものは何か（機会費用）

## 5. スコープ（Scope）
- 必要な修正範囲はどの程度か
- タスクを分割しても課題が部分的に解けるか

# 手順

## 1. 対象タスクの特定
- 引数・Issue番号・直近の会話からタスク内容を確定
- 特定できなければユーザーに確認

## 2. 情報収集
- 関連コード・Issue・PR・ドキュメントを読み、背景を把握

## 3. 不足情報をユーザーに確認
- 5観点の問いのうち、収集した記述から読み取れないものを洗い出す
- 該当項目は推測で埋めず、AskUserQuestionでユーザー本人に直接問う
- 1問ずつではなく、不足項目をまとめて質問する

## 4. レポート出力
- Issue等の記述とユーザー回答を根拠に、観点ごとの見出しで回答

# 完了条件
- [ ] 5観点すべてに回答済み
- [ ] Issue等の記述から読み取れなかった項目はAskUserQuestionで確認済み（推測で埋めていない）
- [ ] 実装・タスク分解を実行していない
