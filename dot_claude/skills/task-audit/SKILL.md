# 役割
着手前タスクを5観点で監査し、着手判断の材料を提示（実装・タスク分解は行わない）

# 前提
- 監査対象は個別タスク・Issue単位
- `$ARGUMENTS`、Issue/PR番号、または直近の会話文脈から対象を特定
- 着手前に実行する事前チェック

# 監査の5観点

各観点は「問い」と、その回答が満たすと懸念になる条件のペア。

## 1. 目的（Purpose）
- 問い: 誰にどんな変化を起こしたいか/何を解決したいか
- 懸念条件: 目的が不明瞭

## 2. アウトカム（Outcome）
- 問い: 何が測定できたら成功と言えるか
- 懸念条件: アウトカムが測定不可能

## 3. 本質（Essence）
- 問い: この課題を一部として構成するより上位の課題は何か/
  課題が成り立つ前提は何か（課題が課題でなくなる場面はあるか）
- 懸念条件: より上位の課題を解決すれば本タスクが不要になる

## 4. 要否（Necessity）
- 問い: やらないとどのロールの何人にどんな影響がいつ起きるか/
  先延ばしの影響・工数・ブロックされるタスクはあるか/
  やることで失うものは何か（機会費用）
- 懸念条件: 要否が低い（やらなくても実質的な影響がない）

## 5. スコープ（Scope）
- 問い: タスクを分割しても課題が部分的に解けるか
- 懸念条件: タスク分割で部分的に解決できる

# 手順

## 1. 対象タスクの特定
- 引数・Issue番号・直近の会話からタスク内容を確定
- 特定できなければユーザーに確認

## 2. 情報収集
- 関連コード・Issue・PR・ドキュメントを読み、背景を把握

## 3. 不足情報をユーザーに確認
- 5観点の問いのうち、収集した記述から読み取れないものを洗い出す
- 該当項目は推測で埋めず、AskUserQuestionでユーザー本人に直接問う
- 1問ずつではなく、不足項目をまとめて質問する

## 4. レポート出力
- Issue等の記述とユーザー回答を根拠に、観点ごとの見出しで回答
- 各観点の懸念条件に該当するものは「懸念」として末尾に箇条書き
  （改善提案は含めず判断材料のみ提示。該当なしなら「なし」）

# 完了条件
- [ ] 5観点すべてに回答済み
- [ ] Issue等の記述から読み取れなかった項目はAskUserQuestionで確認済み
      （推測で埋めていない）
- [ ] 懸念条件に該当する観点を懸念として提示済み
- [ ] 実装・タスク分解を実行していない
