# スペック駆動開発

## 概要

コードを書く前に構造化された仕様書を作成する。
スペックは自分と人間エンジニアの間の唯一の真実の源であり、「何を・なぜ作るか」「完了をどう判断するか」を定義する。
スペックなきコードは当て推量に過ぎない。

## ゲート付きワークフロー

スペック駆動開発は4フェーズ。
現フェーズが検証されるまで次へ進まない。

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 人間       人間      人間       人間
 レビュー   レビュー  レビュー   レビュー
```

### フェーズ1: Specify(仕様化)

高レベルのビジョンから開始。
要件が具体化するまで人間へ質問する。

**前提をただちに明示する。**
スペック本文を書く前に、想定している前提を列挙する。

曖昧な要件を黙って埋めない。
スペックの目的は、コードが書かれる*前に*認識の齟齬を表面化させること。

**以下6つのコア領域を網羅したスペック文書を書く:**

1. **目的(Objective)** — 何を・なぜ作るか。ユーザーは誰か。成功とは何か。
2. **コードスタイル(Code Style)** — 実際のコード片1つが、3段落の説明に勝る。命名規則・整形ルール・良い出力例を含める。
3. **テスト戦略(Testing Strategy)** — テスト配置、カバレッジ期待値、どの関心事にどのテストレベルを充てるか。
4. **境界(Boundaries)** — 3層構造:
   - **常に行う:** コミット前のテスト実行、命名規則の順守、入力検証
   - **事前確認:** DBスキーマ変更、依存追加、CI設定変更
   - **絶対にしない:** 秘匿情報のコミット、vendorディレクトリの編集、承認なしの失敗テスト削除

**スペックテンプレート:**

```markdown
# Spec: [プロジェクト/機能名]

## 目的

[何を・なぜ作るか。ユーザーストーリーや受け入れ基準。]

## コードスタイル

[コード片の例 + 主要規約]

## テスト戦略

[テスト配置、カバレッジ要件、テストレベル]

## 境界

- 常に行う: [...]
- 事前確認: [...]
- 絶対にしない: [...]

## 成功基準

[完了をどう判断するか — 具体的でテスト可能な条件]

## 未解決の問い

[人間の入力が必要な未解決事項]
```

**指示を成功基準に翻訳する。**
曖昧な要件を受け取ったら、具体的な条件へ変換する。

```
要件: 「ダッシュボードを速く」

翻訳した成功基準:
- 4G回線でダッシュボードのLCP < 2.5s
- 初回データ読み込み < 500ms
- 読み込み中のレイアウトシフトなし(CLS < 0.1)
→ この目標値で正しいか?
```

これにより、「速く」の意味を推測せず、明確なゴールへ向けて反復・再試行・問題解決ができる。

生成したスペックはプロジェクトルートに `SPEC.md` として保存し、続行前にユーザーへ確認する。

### フェーズ2: Plan(計画)

検証済みスペックをもとに、技術的な実装計画を生成する。

1. 主要コンポーネントとその依存関係を特定
2. 実装順序を決定(何を先に作るべきか)
3. リスクと緩和策を記載
4. 並行して作れるものと逐次でなければならないものを識別
5. フェーズ間の検証チェックポイントを定義

計画はレビュー可能であること。
人間が読んで「このアプローチで良い」または「Xを変えよ」と言える状態。

### フェーズ3: Tasks(タスク分解)

計画とタスク分解の詳細な手順は `planning-and-task-breakdown` skill に従う。
それが正典。以下はインラインの要約。

計画を離散的で実装可能なタスクへ分解する。

- 各タスクは1回の集中セッションで完了可能
- 各タスクに明示的な受け入れ基準
- 各タスクに検証手順(テスト・ビルド・手動確認)を含む
- タスクは重要度ではなく依存関係で順序付け
- 1タスクで変更するファイルは約5個まで

**タスクテンプレート:**

```markdown
- [ ] Task: [説明]
  - Acceptance: [完了時に真であるべきこと]
  - Verify: [確認方法 — テストコマンド、ビルド、手動確認]
  - Files: [触れるファイル]
```

### フェーズ4: Implement(実装)

タスクを1つずつ実行する。
スペック全体を流し込むのではなく、各ステップで適切なスペック節とソースファイルのみを読み込む。

## スペックを生かし続ける

スペックは一度きりの成果物ではなく、生きた文書。

- **決定が変わったら更新** — データモデル変更が必要と分かったら、まずスペックを更新し、その後実装。
- **スコープが変わったら更新** — 追加・削除された機能をスペックに反映。
- **スペックをコミット** — スペックはコードと共にバージョン管理に置く。
- **PRでスペックを参照** — 各PRが実装するスペック節へリンクを張る。

## よくある言い訳

| 言い訳                                 | 実際                                                                                     |
| -------------------------------------- | ---------------------------------------------------------------------------------------- |
| 「単純だからスペックは不要」           | 単純なタスクに*長い*スペックは不要だが、受け入れ基準は要る。2行のスペックで十分。        |
| 「コードを書いた後にスペックを書く」   | それは仕様化ではなくドキュメント化。スペックの価値はコード前に明確化を強制する点にある。 |
| 「スペックは遅くする」                 | 15分のスペックが数時間の手戻りを防ぐ。15時間のデバッグより15分のウォーターフォール。     |
| 「どうせ要件は変わる」                 | だからこそ生きた文書。古いスペックでも、無いよりはまし。                                 |
| 「ユーザーは欲しいものを分かっている」 | 明確な要求でも暗黙の前提がある。スペックがそれを表面化する。                             |

## レッドフラグ

- 書かれた要件なしにコードを書き始める
- 「完了」の意味を明確にする前に「もう作り始めていい?」と聞く
- どのスペック・タスクにも無い機能を実装する
- 文書化せずにアーキテクチャ上の決定を行う
- 「作るものは自明」としてスペックを飛ばす
