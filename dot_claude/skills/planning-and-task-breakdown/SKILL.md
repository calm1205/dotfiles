# 計画とタスク分解

## 概要

作業を、明示的な受け入れ基準を持つ小さく検証可能なタスクへ分解する。
良いタスク分解は、確実に作業を完了するエージェントと、絡まった混乱を生むエージェントの分かれ目。
各タスクは、1回の集中セッションで実装・テスト・検証できる大きさに収める。

## 使用タイミング

- スペックがあり、実装単位へ分解する必要があるとき
- タスクが大きすぎる・曖昧で着手できないとき
- 複数エージェント・セッションへ作業を並行化するとき
- 人間へスコープを伝える必要があるとき
- 実装順序が自明でないとき

**使わない場合:** スコープが自明な単一ファイル変更、またはスペックに既に明確なタスクが含まれるとき。

## 計画プロセス

### ステップ1: プランモードに入る

コードを書く前に、読み取り専用モードで動く。

- スペックと関連コードベースを読む
- 既存のパターン・規約を特定
- コンポーネント間の依存を把握
- リスクと未知を記録

**計画中はコードを書かない。**
成果物は `tasks/plan.md` に保存する計画文書と `tasks/todo.md` に保存するタスクリストであり、実装ではない。

### ステップ2: 依存グラフを特定

何が何に依存するかを把握する。

```
Database schema
    │
    ├── API models/types
    │       │
    │       ├── API endpoints
    │       │       │
    │       │       └── Frontend API client
    │       │               │
    │       │               └── UI components
    │       │
    │       └── Validation logic
    │
    └── Seed data / migrations
```

実装順序は依存グラフのボトムアップ。
まず基盤を作る。

### ステップ3: 垂直にスライスする

DBを全部、次にAPIを全部、次にUIを全部——ではなく、一度に1つの完結した機能パスを作る。

**悪い例(水平スライス):**
```
Task 1: データベーススキーマ全体を構築
Task 2: 全APIエンドポイントを構築
Task 3: 全UIコンポーネントを構築
Task 4: 全体を接続
```

**良い例(垂直スライス):**
```
Task 1: ユーザーがアカウント作成できる(登録のschema + API + UI)
Task 2: ユーザーがログインできる(認証schema + API + UI)
Task 3: ユーザーがタスク作成できる(タスクschema + API + UI)
Task 4: ユーザーがタスク一覧を見られる(query + API + 一覧UI)
```

各垂直スライスが、動作しテスト可能な機能を届ける。

### ステップ4: タスクを書く

各タスクは以下の構造に従う。

```markdown
## Task [N]: [短い説明的なタイトル]

**Description:** このタスクが達成することを1段落で説明。

**Acceptance criteria:**
- [ ] [具体的でテスト可能な条件]
- [ ] [具体的でテスト可能な条件]

**Verification:**
- [ ] テスト通過: `npm test -- --grep "feature-name"`
- [ ] ビルド成功: `npm run build`
- [ ] 手動確認: [確認内容]

**Dependencies:** [依存するタスク番号、または "None"]

**Files likely touched:**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**Estimated scope:** [Small: 1-2 files | Medium: 3-5 files | Large: 5+ files]
```

### ステップ5: 順序付けとチェックポイント

以下を満たすようタスクを並べる。

1. 依存が満たされる(基盤を先に作る)
2. 各タスクがシステムを動作状態に保つ
3. 2〜3タスクごとに検証チェックポイントを置く
4. 高リスクタスクを早期に置く(早く失敗する)

明示的なチェックポイントを追加する。

```markdown
## Checkpoint: Task 1-3 の後
- [ ] 全テスト通過
- [ ] アプリがエラーなくビルド
- [ ] コアのユーザーフローがE2Eで動作
- [ ] 続行前に人間とレビュー
```

## タスクサイズの指針

| サイズ | ファイル | スコープ | 例 |
|------|-------|-------|---------|
| **XS** | 1 | 単一関数・設定変更 | 検証ルール追加 |
| **S** | 1-2 | 1コンポーネント・1エンドポイント | 新規APIエンドポイント追加 |
| **M** | 3-5 | 1機能スライス | ユーザー登録フロー |
| **L** | 5-8 | 複数コンポーネントの機能 | フィルタ・ページング付き検索 |
| **XL** | 8+ | **大きすぎ — さらに分解する** | — |

L以上のタスクは、より小さく分解する。
エージェントはS・Mタスクで最も良く機能する。

**さらに分解すべきサイン:**
- 1回の集中セッションを超える(概ねエージェント作業2時間以上)
- 受け入れ基準を3項目以下で書けない
- 2つ以上の独立サブシステムに触れる(例: 認証と課金)
- タスクタイトルに「〜と〜」と書いている(2タスクの兆候)

## 出力ファイル

- **計画文書:** 実装計画を `tasks/plan.md` に保存。
- **タスクリスト:** チェックリスト形式のタスクリストを `tasks/todo.md` に保存。

`tasks/` が無ければ作成する。
これらのパスは `/build` コマンド等の下流ツールが前提とする規約。

## 計画文書テンプレート

```markdown
# Implementation Plan: [機能/プロジェクト名]

## Overview
[何を作るかの1段落要約]

## Architecture Decisions
- [主要な決定1と根拠]
- [主要な決定2と根拠]

## Task List

### Phase 1: Foundation
- [ ] Task 1: ...
- [ ] Task 2: ...

### Checkpoint: Foundation
- [ ] テスト通過・ビルドクリーン

### Phase 2: Core Features
- [ ] Task 3: ...
- [ ] Task 4: ...

### Checkpoint: Core Features
- [ ] E2Eフローが動作

### Phase 3: Polish
- [ ] Task 5: ...
- [ ] Task 6: ...

### Checkpoint: Complete
- [ ] 全受け入れ基準を満たす
- [ ] レビュー準備完了

## Risks and Mitigations
| Risk | Impact | Mitigation |
|------|--------|------------|
| [リスク] | [High/Med/Low] | [戦略] |

## Open Questions
- [人間の入力が必要な問い]
```

## 並行化の機会

複数エージェント・セッションが利用可能なとき。

- **並行化して安全:** 独立した機能スライス、実装済み機能のテスト、ドキュメント
- **逐次が必須:** DBマイグレーション、共有状態の変更、依存チェーン
- **調整が必要:** APIコントラクトを共有する機能(先にコントラクトを定義し、その後並行化)

## よくある言い訳

| 言い訳 | 実際 |
|---|---|
| 「進めながら考える」 | それが絡まった混乱と手戻りの元。10分の計画が数時間を節約する。 |
| 「タスクは自明」 | それでも書き出す。明示的なタスクが隠れた依存や見落としたエッジケースを表面化する。 |
| 「計画はオーバーヘッド」 | 計画がタスクそのもの。計画なき実装はただのタイピング。 |
| 「全部頭に入る」 | コンテキストは有限。書かれた計画はセッション境界と圧縮を越えて残る。 |

## レッドフラグ

- 書かれたタスクリストなしに実装を始める
- 受け入れ基準なく「機能を実装する」とだけ書くタスク
- 計画に検証ステップがない
- 全タスクがXLサイズ
- タスク間にチェックポイントがない
- 依存順序が考慮されていない

## 検証

実装を始める前に確認する。

- [ ] 各タスクに受け入れ基準がある
- [ ] 各タスクに検証ステップがある
- [ ] タスク依存が特定され、正しく順序付けられている
- [ ] どのタスクも約5ファイルを超えない
- [ ] 主要フェーズ間にチェックポイントがある
- [ ] 人間が計画をレビュー・承認済み

## 関連

受け入れ基準はタスクごとで「正しいものを作ったか」に答える。
その土台に、全タスクが満たすプロジェクト共通の Definition of Done がある。
`references/definition-of-done.md` を参照。
