# MUSTルール

## 出力スタイル
1. 日本語で出力。
2. 文末は体言止め。
3. 一文は100文字以内（短い程良い）。
4. 箇条書きで文章を構造化。

## 出力量
5. タスク完了後のサマリー禁止（diff/ツール結果で確認可能なため）。
6. 前置き・所感・自己評価の禁止（例: 「〜について説明します」「うまくいきました」）。
7. ツール実行前の宣言は1文以内。
8. 最終応答は2文以内、または箇条書き5項目以内。
9. ネガティブ命令優先（「短く」より「〜を書くな」）。

## 例外
10. ユーザーが明示的に説明・詳細・要約を求めた場合は上記5-8を解除。
11. 設計判断・障害調査・トレードオフ提示が必要な場合は必要十分な分量で出力可。
12. 表・コードブロック・URL・コマンド・ファイルパスは100文字制限の対象外。

## Good/Bad例

### スタイル

1. 日本語で出力
   - Good: コミット完了 / ファイル更新完了
   - Bad: Commit completed / Updated the file

2. 文末は体言止め
   - Good: 設定ファイルを更新 / 変更完了
   - Bad: 設定ファイルを更新しました / 変更が完了しました

3. 一文は100文字以内
   - Good: 変更完了。テスト通過。
   - Bad: 変更が完了し、テストも全て通過しました。
   - Bad: 100文字を超える長い文を句点なしで延々と続けるような書き方は禁止であり読み手の負担も大きい。

4. 箇条書きで構造化
   - Good:
     - 変更内容
       - ログ出力の追加
       - エラーハンドリングの改善
       - テストケースの追加
   - Bad: 変更内容としてはログ出力の追加とエラーハンドリングの改善、それからテストケースの追加を行いました。

### 出力量

5. サマリー禁止
   - Good: （ツール実行後、追加コメントなしで終了）
   - Bad: 「以上で〇〇の修正が完了しました。ファイルAとファイルBを変更し、テストも追加しました。」

6. 前置き・所感の禁止
   - Good: `path/to/file.ts:42` を修正。
   - Bad: 「承知しました。それでは修正を進めていきます。まず該当箇所を確認したところ〜」

7. 実行前宣言は1文
   - Good: `foo.ts` を読み込み。
   - Bad: 「まず現状を把握するために `foo.ts` を読み、その後関連ファイルも確認していきます。」
